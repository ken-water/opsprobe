# OpsProbe v0.10.4

Release date: 2026-06-08

`v0.10.4` improves the desktop first-run experience by turning post-install health checks into an operator-facing readiness view with grouped repair guidance and stronger UI regression coverage.

## Highlights

- Adds a readiness summary to `System Settings` so users can tell immediately whether the desktop is ready for the first real inspection
- Groups low-level health checks into repair packs for runtime, report exports, SSH tooling, and recurring inspections
- Extends browser UI coverage so first-run readiness and repair entry points are now part of the desktop release gate

## Why this release exists

`0.10.3` made runtime recovery and machine-move trust more explicit, but the post-install experience still expected operators to interpret raw checks and scattered guidance. `0.10.4` closes that gap by making the desktop explain what matters, what is blocked, and what action should happen next.

## Validation

- `npm --workspace @opsprobe/desktop run typecheck`
- `npm run test:desktop-browser-ui`

## Known Limits

- The desktop is clearer about local repair paths, but it still depends on the operator to perform host-level installs, PATH fixes, and permission repair outside the app
- Managed runtime actions still rely on best-effort local process control rather than a hardened platform-native supervisor
- `0.10.4` improves first-run clarity, but clean-machine install and broader stable-candidate evidence still remain for the later pre-stable line
