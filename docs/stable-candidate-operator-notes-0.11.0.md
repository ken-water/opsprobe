# Stable Candidate Operator Notes: 0.11.0 Draft

This draft is based on the current `0.11.0` evidence line and the latest clean-user-profile validation artifacts.

It is not the final `1.0.0` operator sign-off. Its purpose is to turn the current evidence into reviewable operator-facing conclusions before Issue `47` is resumed.

## Candidate Metadata

- Candidate version: `0.11.0`
- Validation date: `2026-06-08`
- Validator: Codex-assisted repository validation
- Machine or VM label: local development environment
- Operating system: local Linux workspace
- PostgreSQL binaries available in default `PATH`: no
- PostgreSQL binaries discoverable by OpsProbe runtime: yes (`/usr/lib/postgresql/16/bin`)
- Validation home/profile path: temporary clean `HOME` generated by `./scripts/validate-clean-user-profile.sh`
- Managed PostgreSQL validation port: temporary `OPSPROBE_POSTGRES_PORT` chosen by the validation script to avoid same-machine port collisions
- Desktop stable-candidate validation command: `./scripts/validate-desktop-stable-candidate.sh`
- Desktop operator walk-through command: `./scripts/validate-desktop-operator-walkthrough.sh`
- Desktop bundle-candidate validation command: `./scripts/validate-desktop-bundle-candidate.sh`

## 1. Install And Bootstrap

### Local prerequisites seen by the operator

- Node / npm: present
- Rust / cargo: present
- Tauri dependencies: build path passed during recent release validation, but this draft did not re-run desktop packaging directly
- PostgreSQL binaries: not present in the default shell `PATH`, but successfully auto-discovered from `/usr/lib/postgresql/16/bin`

### First-run status result

- Command used: `HOME="<tmp>" npm run local-service:status`
- `~/.opsprobe` was created automatically under the temporary profile
- Missing prerequisites were surfaced through status checks instead of failing silently
- `recoveryActions` appeared and gave a next-step path even when PostgreSQL binaries were unavailable

### Bootstrap notes

- Command used:
  - `HOME="<tmp>" npm run local-service:postgres-bootstrap`
  - `HOME="<tmp>" npm run local-service:postgres-start`
  - `HOME="<tmp>" npm run local-service:status`
- In the current captured environment, managed PostgreSQL bootstrap and start were proven from a clean profile without manually exporting PostgreSQL bin paths
- Status moved to `ready`, the managed PostgreSQL process became healthy, and the storage backend switched to PostgreSQL successfully
- The validation script now uses a temporary `OPSPROBE_POSTGRES_PORT` so the same operator can repeat the proof on one machine without colliding with prior local runs

## 4. Stop / Restart / Recovery

### Stop flow

- Command or UI path: `npm run local-service:stop`
- The result is explicit and no longer relies on implicit PID assumptions
- The current behavior is honest enough to review for `1.0.0`, even if supervision remains best-effort

### Restart flow

- Command or UI path: `node --experimental-strip-types ./apps/local-service/src/main.ts restart`
- The product does not claim the background process is already healthy after restart preparation
- The next step remains explicit: start the service again and review recovery guidance

### Recovery guidance

- Current captured `recoveryActions` explain start-or-restart, PostgreSQL bootstrap, and other repair steps depending on environment state
- This is a meaningful improvement over earlier opaque status behavior
- For `1.0.0`, this area looks documentable, but still not equivalent to hardened service-manager supervision

List anything that still blocks a credible stable release:

- Blocker 1: current evidence is still mostly repository-driven; desktop build credibility is better now, but a packaged desktop-oriented stable candidate walk-through is still not captured strongly enough

List any limit that might be acceptable for `1.0.0` if documented honestly:
